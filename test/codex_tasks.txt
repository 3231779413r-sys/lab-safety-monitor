Codex 修改任务清单
====================

更新时间：2026-07-01

目标
----
围绕“命名统一、画面预览、检测推理”三条主线，给出可执行优化清单，优先解决误报、卡顿、传输延迟和链路不一致问题。

工作要求
--------
1. 先审阅并理解以下关键文件，再开始改：
- `backend/app/core/danger_events.py`
- `backend/app/ml/yolov11_detector.py`
- `backend/app/ml/temporal_filter.py`
- `backend/app/services/persistence.py`
- `backend/app/api/routes/cameras.py`
- `frontend/src/lib/formatters.ts`
- `frontend/src/app/(authenticated)/cameras/page.tsx`
- `frontend/src/app/(authenticated)/events/page.tsx`
- `frontend/src/app/(authenticated)/persons/page.tsx`

2. 修改时遵守这些原则：
- 先统一 canonical key，再做别名兼容
- 不要破坏现有 API/前端接口
- 预览链路优先降频、缓存和节流
- 推理链路优先减少误报和尾延迟
- 所有改动尽量局部、可回滚
- 改完后给出验证步骤和建议配置

3. 重点问题背景（需要针对性修复）：
- 现在存在 hardhat / hard_hat、safety_vest / vest、work_clothes 等命名不一致
- 前端筛选、后端归一化、历史数据标签存在混用风险
- 预览请求频率偏高，多个页面容易同时拉同一帧，造成带宽和 CPU 压力
- 推理日志里有持续的 frame timing 波动，说明瓶颈更像链路抖动和队列等待，不是单纯模型计算
- temporal filter 的按类型连续帧门槛需要和 buffer 语义一致
- dataset/存储型后处理可能把主推理链路拖长

P0：必须先完成
================

任务 P0-1：统一命名和别名映射
目标：
- 确保后端 canonical key、前端筛选项、历史数据展示、告警标签完全一致
- 保留旧 key 兼容，但新写入只使用 canonical key
- 避免同一违章在不同页面显示成多个名字

重点修改位置：
- `backend/app/core/danger_events.py`
- `backend/app/ml/yolov11_detector.py`
- `backend/app/ml/temporal_filter.py`
- `backend/app/services/persistence.py`
- `frontend/src/lib/formatters.ts`
- `frontend/src/app/(authenticated)/cameras/page.tsx`
- `frontend/src/app/(authenticated)/events/page.tsx`
- `frontend/src/app/(authenticated)/persons/page.tsx`

具体要求：
- 统一 hardhat / hard_hat 只保留一个 canonical key
- 统一 safety_vest / vest / no_vest 的展示和存储口径
- 统一 work_clothes 的标签和筛选 key
- 增加命名回归测试，防止前后端再次分叉

任务 P0-2：降低预览链路卡顿和重复拉取
目标：
- 预览页面不要高频重复请求同一帧
- 多个页面或组件同时访问时要命中缓存而不是重复读源头
- 出错回退时不要反复生成高成本 placeholder frame

重点修改位置：
- `backend/app/api/routes/cameras.py`
- `frontend/src/app/(authenticated)/cameras/page.tsx`
- 需要的话，连同其它预览入口一起检查

具体要求：
- 预览请求按 raw / annotated / people / status 分层统计
- 给预览读取加短 TTL 或最近帧缓存
- 前端拆分列表刷新和预览刷新
- 对高频失败路径加节流，避免 CPU 被 placeholder 生成吃掉

任务 P0-3：修正检测推理链路的误报和尾延迟
目标：
- 减少“正类和反类同时出现”导致的误报
- temporal filter 的按类型门槛必须与 buffer 语义一致
- 推理结果在进入存储/持久化前尽量先稳定归一

重点修改位置：
- `backend/app/ml/yolov11_detector.py`
- `backend/app/ml/temporal_filter.py`
- `backend/app/services/persistence.py`

具体要求：
- 保证正检优先，冲突时抑制反检
- 对 hardhat 等高门槛事件补足稳定判定
- dataset 或其它附加上传逻辑不要拖慢主推理路径

P1：性能与稳定性增强
====================

任务 P1-1：优化预览缓存和刷新策略
目标：
- 多个页面同时查看时，避免重复拉取同一帧
- 提高预览命中缓存概率，减少带宽和 CPU 压力
- 让前端预览节奏更平滑，不因短时抖动频繁刷帧

重点修改位置：
- `backend/app/api/routes/cameras.py`
- `frontend/src/app/(authenticated)/cameras/page.tsx`
- `frontend/src/app/(authenticated)/events/page.tsx`
- `frontend/src/app/(authenticated)/persons/page.tsx`

任务 P1-2：平滑检测门槛和稳定判定
目标：
- 让 temporal filter 的按类型连续帧门槛与 buffer 一致
- 降低硬帽、背心等高频误报
- 让稳定违章更符合连续帧语义，而不是偶发噪声

重点修改位置：
- `backend/app/ml/temporal_filter.py`
- `backend/app/ml/yolov11_detector.py`

任务 P1-3：解耦存储型后处理
目标：
- dataset/额外上传类动作不要拖慢主推理链路
- 必要时改成异步、采样或事件触发

重点修改位置：
- `backend/app/services/persistence.py`
- `backend/app/services/object_storage.py`

任务 P1-4：增强运行时可观测性
目标：
- 让日志更容易定位卡在预览、推理还是存储
- 至少保留：预览耗时、推理耗时、持久化耗时、失败回退耗时

重点修改位置：
- `backend/app/api/routes/cameras.py`
- `backend/app/services/persistence.py`

P2：验证与部署建议
==================

修改完成后请做这些事：
1. 针对命名、预览和推理链路分别做最小范围验证。
2. 如果项目里没有现成测试，就至少做静态检查、启动路径检查或关键函数级验证。
3. 输出结果总结，包含：
   - 命名是否已统一
   - 预览是否已降频/缓存
   - 推理误报和尾延迟是否下降
   - 还存在哪些未覆盖风险
4. 最后给出一份可直接复制给 Codex 的后续优化指令。
- 改了哪些文件
- 每项修改解决了什么问题
- 还存在哪些未解决风险
- 推荐的 `.env.docker` 参数组合
- 如何观察是否真的达到“多相机实时处理最新帧”的目标

额外要求
--------
- 不要只做配置调整，必须修改代码逻辑
- 不要只改日志文案，必须改调度和可靠性
- 不要删除现有功能（人脸、PPE、事件持久化、异步身份回填都要保留）
- 如果某个点需要折中，请在最终总结里明确说明 trade-off

后续小任务（2026-06-24 二次复查）
-----------------------------
1. 确认所有 `monitor-worker` / `worker-shard-*` 进程已经完整运行新代码版本。
   - 代码里已经新增 `effective_processed_fps`、`latest_frame_age_ms`、`replaced_pending_count`、`submit_queue_reject_count`、`identity_skip_ratio`、`async_identity_merge_latency_ms`、`camera_hotness_score`
   - 但实际 `CAMERA_RUNTIME_SUMMARY` 日志里很多相机仍未输出这些字段
   - 需要排查是容器未完全重启、旧进程仍在运行，还是日志路径里混有旧实例输出
   - 目标：确保新的 summary 字段真实落到日志中，便于后续判断“是否真的处理最新帧”

2. 针对少数热点相机做点状优化，不做系统级大改。
   - 优先检查高延迟或高 backpressure 的点位
   - 当前复查中可重点关注：
     - `c7bb5a0f-946d-41ef-bfa5-74405363a229`
     - `76004f54-ef00-465f-921b-c86bc8cc8490`
   - 检查项包括：
     - 是否应固定为 `fast` profile
     - 是否使用了过高分辨率/主码流
     - 场景内人数或复杂度是否长期偏高
     - 是否需要单独迁移到负载更低的 shard
   - 目标：降低个别热点相机的 P95 推理时延，而不改动当前整体稳定链路

3. 专项修复：当前版本“检测有了，但卡顿仍明显”，允许继续修改，但必须限制修改范围，重点解决不合理参数与热点相机策略问题。
   背景：继续复查后确认，当前版本的主 worker 调度已经比之前稳定，但实时性仍然不理想：
   - `inference_total_ms` 平均约 `340.9ms`，P95 约 `863ms`，P99 约 `1152ms`
   - `latest_frame_age_ms` 平均约 `763ms`，P95 约 `1293ms`，P99 约 `1729ms`，最大超过 `4.5s`
   - 热点相机集中明显，并非全局平均问题
   - 当前更像是“结果能出来，但很多结果已经偏旧”，导致前端肉眼感受仍然卡
   这说明问题已经不是“复杂调度把链路搞坏”，而是当前配置仍然过于偏向“保结果”，不够偏向“保最新帧”。

   本轮目标：
   - 继续解决卡顿
   - 优先压低 `latest_frame_age_ms`
   - 优先改善前端实时观感
   - 保持现有检测能力，不回退到“几乎没人被检测”的状态

   修改原则：
   - 可以继续修改，不要因为怕改坏就不敢改
   - 但修改必须合理、可解释、低风险
   - 本轮重点改参数和热点相机策略，不再大改 worker 主调度逻辑
   - 不允许重新引入复杂 `fast_lane` / `complex_lane` / `force_dispatch_stale` 体系

   重点方向：
   1. 调整参数，恢复“最新帧优先”
      建议优先尝试：
      - `CAMERA_RUNTIME_LATEST_ONLY_MODE=true`
      - `INFERENCE_MAX_PENDING_FRAMES_PER_CAMERA=1`
      - `INFERENCE_BATCH_SIZE=2`
      - `INFERENCE_MIN_BATCH_SIZE=1`
      - `INFERENCE_MAX_BATCH_SIZE=2`
      - `INFERENCE_BATCH_TIMEOUT_MS=6`
      原因：
      - 当前 `latest_only=false`、`pending=4` 明显更容易积累旧帧
      - 当前 batch 仍偏大，不利于压低 frame age

   2. 对热点相机单独降载，而不是所有相机一刀切
      当前热点相机重点关注：
      - `37efcd63-7042-4d3b-bafd-fa32b9444d96`
      - `926d2c74-d2fa-4bd7-84da-7a3646f2196e`
      - `f082b8a4-acbb-4acd-8e8a-9e54371d671f`
      - `63518f35-649e-4804-b200-c367ef24f85c`
      - `72f58fa2-8406-4520-b57d-fca20ad19c92`
      - `7857c36a-ad73-41e3-95f2-3e96053f8e37`
      对这些相机优先考虑：
      - 固定 `fast` profile
      - 必要时单独覆盖配置
      - 目标是先压住这些相机的 `latest_frame_age_ms`

   3. 代码只允许小修，不允许重构主流程
      如需改代码，只允许这类修改：
      - runtime 侧更积极地保最新帧
      - 热点相机配置覆盖能力增强
      - summary / metrics 更准确反映热点状态
      - 轻微修正 batch 参数读取或应用方式
      禁止：
      - 重写 `backend/app/inference_batch_worker.py` 主调度框架
      - 新增复杂 lane 系统
      - 用新的 stale/priority 机制替代当前简单逻辑

   本轮验收标准：
   - `latest_frame_age_ms` 明显下降
   - `inference_batch_wait_ms` / `avg_wait_ms` / `p95_wait_ms` 明显下降
   - 前端肉眼卡顿感明显减轻
   - 热点相机的旧帧问题明显缓解
   - 保持持续有人体检测结果，不回退到“几乎没人被检测”的状态

   输出要求：
   - 明确列出最终采用的参数组合
   - 明确列出哪些热点相机被单独降档/覆盖
   - 给出修改前后对比：
     - `latest_frame_age_ms`
     - `inference_total_ms`
     - `avg_wait_ms/p95_wait_ms`
     - 热点相机表现
   - 如果按上述调参后仍然明显卡顿，请明确说明：当前 29 路是否已经超过现有 inference worker / shard 容量极限，不要继续靠代码硬扛

4. 日志与部署一致性收尾。
   - 统一 `CAMERA_RUNTIME_SUMMARY` / `CAMERA_RUNTIME_SHARD_SUMMARY` 的输出与检索方式
   - 确认所有 worker/shard 都运行同一版代码
   - 如果日志目录中混有旧格式输出，请在总结里明确说明，避免误判
